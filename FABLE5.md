# FABLE5.md — 페이블5의 뇌로 생각하기 (착수 사고 프로토콜)

> **역할**: 작업 지시를 받으면 **답하기 전에 계획하고, 가정하고, 스스로 검증**하게 만드는 강제 프로토콜.
> CLAUDE.md §1.5가 이 파일을 참조한다. **핵심 = 출력 의무: 아래 블록이 응답에 안 보이면 프로토콜을 안 한 것이다.**

## 발동 기준 (차등 — 과잉 방지)

| 작업 규모 | 발동 |
|---|---|
| **대형**: 신규 구현·실험 설계/실행·문서/덱 제작·아키텍처 결정·다파일 수정 | **전체 블록 출력 필수** |
| **중형**: 단일 파일 수정·분석·재실행 | 축약 1~3줄 (핵심 가정+깨질 것) |
| 소형: 질문 답변·1줄 수정·조회 | 생략 가능 (단, 위험 감지 시 승격) |

## 사전점검 블록 (대형 작업 — 응답 첫 부분에 출력)

```
🧠 사전점검
- 요구 재정의: <지시를 내 말로 1문장 — 오해 소지 있으면 여기서 질의>
- Blind Spot: <내가 놓치고 있을 것·위험한 가정 1~3개>
- 깨질 것: <성급히 진행하면 무엇이 깨지나 1~2개>
- 최소 슬라이스: <가장 작은 검증 가능 단위>
- 검증 계획: <완료를 무엇으로 증명하나 (게이트·렌더·실측)>
- 적용 스킬/회상: <매칭되는 스킬·기억 회상 필요 여부 — 없으면 "없음">
```

- **요구 재정의에서 모호점이 나오면 착수 전에 질의** — 단, 가역적·소규모면 [가정] 기록 후 진행(가역성 규칙)
- **위험한 가정은 implementation-notes.md에 [가정](근거+위험도)으로 선기록** — 블록에만 쓰고 버리지 말 것
- 검증 계획은 작업 후 실제 실행 — "계획만 쓰고 검증 생략"이 최다 실패 모드

## 5단계 검증 게이트 (범위→근거→풀기→검증→보고)

> 사전점검 블록이 ①의 입구라면, 이 절은 **실행 중간·종료의 관문**이다. 답하기 전 이 관문을 거친다 — 실력의 절반은 꼼꼼한 일하는 방식에서 나온다. 체크 항목·표준 증거 카탈로그 정본: [Docs/reviewer-checklist.md](Docs/reviewer-checklist.md)

| 단계 | 게이트 (통과 기준) | 비고 |
|---|---|---|
| ① 범위 | 사전점검 블록 = 이 게이트. 모호점 잔존 시 착수 금지 | 기존 |
| ② 근거 | 핵심 주장마다 근거유형 태깅: `file:line`/실측로그/사용자확정/**미확인**. 미확인으로 ③ 진입 시 [가정] 선기록. 수치 합성 경로의 이중 기여 점검 | **신설** |
| ③ 풀기 | 최소 diff·1수정 1재검증·가역성 판단 (기존 규칙) | 기존 |
| ④ 검증 | 완료 선언 전 검증 명령 **실제 실행+출력 인용**. 표준 증거 카탈로그(checklist)에서 선택. 기동 시그니처만 믿지 말고 실물 진행(예: sim time)까지 | **신설** |
| ⑤ 보고 | 결과+한계 병기, 증거 ID 필수, 미검증="미확인" 명시, 실패도 그대로 | **신설** |

**차등 (발동 기준 표와 동일 축)**: 대형 = 5게이트 전부 + 말미 `✅ 게이트: ①~⑤ 통과 (증거: …)` 1줄 / 중형 = ②④⑤ 축약 / 소형 = 생략 / **반복 정형 작업(시뮬 재시작 등) = ④만**(시그니처 확인). 대형·비가역·크로스머신 규격 변경은 L2 적대검증 서브 추가 (checklist §L2).

## 사고 원칙 (블록 작성 시의 기준)

1. **정본 우선, 추정 금지** — 수치·절차는 정본 확인 후. 없으면 "미확인" 명시, 발명 금지
2. **관찰이 정적 검사를 이긴다** — 실물(렌더·로그·실행)에서 본 것만 "된다"고 말한다
3. **경쟁 가설 2개 이상** — 원인 조사·이상 관찰 시 단일 가설로 달리지 않는다
4. **최소 diff·최소 슬라이스** — 큰 것을 한 번에 만들지 않는다
5. **1수정 1재검증** — 고친 것이 새로 깨뜨린 것을 확인한다

## 스킬·기억 활용 판단 (사전점검의 "적용 스킬/회상" 항목)

**먼저 목적 기준으로 생각하라**: 지시의 표면 형태(시킨 방식 그대로)가 아니라 *목적*에 가장 적합한 수단이 무엇인지 스스로 판단한다 — **표에 없는 더 나은 스킬·행동·작업 방식이 있으면 그것을 택하고(필요하면 제안하고), 표는 출발점일 뿐이다.** 예: "차트 그려줘"의 목적이 발표 설득이면 차트보다 대비 표가 나을 수 있고, "수정해줘"가 반복될 조짐이면 수정 대신 자동화가 답일 수 있다.

아래는 **강제 규칙이 아니라 판단 재료(카탈로그)**다: *"이 스킬/회상이 실제로 결과를 바꾸는가, 오버헤드인가"* — 가치가 있을 때만 호출하고, 판단 결과(선택한 수단과 이유)를 블록의 "적용 스킬/회상" 줄에 남긴다. 스킬명 발명 금지(로드된 것만):

| 작업 신호 | 스킬/도구 |
|---|---|
| 새 기능·설계 구상("만들자","어떻게 할까") | superpowers:brainstorming — **계획 착수 전** |
| 버그·테스트 실패·원인 불명 | superpowers:systematic-debugging (재현→경쟁 가설→증거) |
| 구현·설계·리팩터 착수 전 | **memoir:memory-recall** — 과거 결정·선호·함정 회상 (관련 기억이 접근을 바꿈) |
| 발표 덱 — **시각 품질 우선**(발표·데모·제출) | **HTML/CSS 직접 코딩 → 헤드리스 Chrome 2배 스케일 스크린샷 → JPEG→PDF(+이미지 pptx)** — 타이포·레이아웃 자유, 겹침류 결함 구조적 회피 (기아 PT 실증) |
| 발표 덱 — **수신자 편집 필요**·문서 파일(docx/pdf) | document-skills:pptx(python-pptx 조립) — QA 루프(렌더→육안→수정) 필수 |
| 도식·이미지 생성 | 정밀 라벨=matplotlib 직접 / 일러스트형=codex-image(병렬 5장) |
| 2단계 이상 연속 작업 | goals.py 체크포인트 (fablize) |
| 대규모 병렬·다관점 검증 | Workflow (사용자 명시 시) / Agent 병렬 |
| 한글 문체 다듬기 | humanize-korean 계열 |

판단 기준: ①결과를 실제로 바꾸는가(품질·안전·속도) ②작업 규모 대비 오버헤드가 정당한가 ③과거에 이 유형에서 스킬이 도움됐나. 참고: 기억 회상은 비용이 가장 싸고(1회), 놓쳤을 때 손실이 커서 구현·설계 착수에는 대체로 값어치가 있다 — 그래도 최종 판단은 상황 기준.

## 서브에이전트·경량 모델에 이식

병렬 에이전트를 띄울 때 이 프로토콜의 핵심을 프롬프트에 1줄로 주입한다:
> "착수 전 놓친 것·위험한 가정·깨질 것을 먼저 나열하고, 완료 전 자기 검증(게이트/렌더 재확인)을 수행하라. 확인 못 한 것은 발명하지 말고 '미확인'으로 표기하라."

## 자기 감사 (주기적)

세션 종료 배치 기록 때 1회: 이 세션의 대형 작업 중 사전점검 블록 없이 착수한 것이 있었나 — 있으면 그 자체를 implementation-notes에 [이탈]로 기록.

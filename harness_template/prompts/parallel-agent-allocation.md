# ⑩ 병렬 에이전트 할당 설계

**언제 쓰나**: 작업을 쪼개 병렬 에이전트/워크플로우로 돌릴 때 — 할당 구조 자체를 설계하는 단계. (워크플로우는 사용자 명시 시)

## 할당 패턴 (작업 유형 → 구조)

| 작업 유형 | 할당 구조 |
|---|---|
| 주요 구현 | **구현+기록 / 설계검토 2병렬** (기본) |
| 대상이 넓은 추출·명세화 | **도메인 분할 N병렬 매핑 → 합성 1** <!-- 예: ICD 4도메인(체화/전략/집단/기반)→합성 --> |
| 신뢰가 필요한 산출물 | **제작 → 독립 QA(신선한 눈) → 수정 → 재검** — 제작자 자체 QA만 믿지 않기 |
| 문서·수치 대량 검증 | 파일/섹션 분할 병렬 감사 (⑦ 카드) |
| 확정 전 주장·설계 | 적대검증 렌즈 2+ 병렬 (② 카드) |
| 장기 실행(비행·빌드·대량 런) | 백그라운드 + 메인은 병행 작업, 완료 통지로 회수 |

## 워크플로우 4페이즈 골격 (규모 큰 제작·명세화의 표준)

```
Phase 1 추출/매핑 (N병렬): 도메인별 담당 — "코드/정본 실물 확인, 발명 금지, 근거 파일:줄 병기"
Phase 2 합성/작성 (1~2): 추출 결과만 사용해 산출물 작성 — 새 내용 발명 금지
Phase 3 적대검증 (2렌즈 병렬, findings 스키마): 산출물의 구체 주장을 실물과 재대조 — "문제없는 항목은 보고 금지"
Phase 4 수정 (조건부): 확정 결함만 반영 + 버전 이력 기록
```

## 에이전트 프롬프트 공통 주입 (모든 할당에)

```
착수 전 놓친 것·위험한 가정·깨질 것을 먼저 나열하고, 완료 전 자기 검증(게이트/렌더 재확인)을 수행하라.
확인 못 한 것은 발명하지 말고 '미확인'으로 표기하라. 근거는 파일:줄로.
```

원칙: ①에이전트 보고는 오케스트레이터가 핵심만이라도 직접 재검증 후 채택 ②병렬 간 파일 충돌 시 worktree 격리 ③백그라운드는 출력 파일 추적 필수(/dev/null 금지).

**실증**: ICD v0.1(8에이전트, 검증이 결함 1건만 — 매핑 품질 증명), 하네스 템플릿(7에이전트, 착수 시뮬레이션이 11건 발굴), 주간 덱(제작→독립 QA 이중 통과), MCR 아티팩트 분석(백그라운드 분석+메인 실기 병행).

# ① 하니스/코드 신설→경화 워크플로우

**언제 쓰나**: 새 실행 스크립트·러너·하니스를 만든 직후, 본 실행(비용 큰 런) 전에 결함을 선제 제거할 때.

## 프롬프트

```
{{신설 코드/하니스 경로}}를 본 실행 전에 경화하라. 절차:
1. 다(多)렌즈 리뷰 — 서로 다른 관점의 검토 에이전트를 병렬로 띄워
   (정확성 / 실패모드 / 환경·설정 누락 / 종료·정리 경로) 각자 결함 후보를 수집
2. 적대 검증 — 각 후보를 "실제로 깨지는 시나리오"로 반박 시도,
   확정(CONFIRMED)만 남기고 과잉 지적은 기각 근거와 함께 버린다
3. 확정 건 전건 수정 — critical부터. 수정이 기존 지표/출력 계약을
   바꾸는지 여부를 명시 (바꾸면 사용자 질문)
4. 게이트 재실행 — {{게이트 목록: 문법검사·단위테스트·selftest·dry-run}}
   전건 PASS 증거를 남긴 뒤에만 "경화 완료" 선언
   <!-- 예: 셸 7종·py 10종·selftest 10·dry-run 5 -->
본 실행은 게이트 PASS 후 별도 승인으로만.
```

**실증**: 시뮬레이션 실행 하니스 경화(64에이전트) — 결함 17건 확정·전건 수정(critical 3: 환경변수 누락·tick 공급원 부재·INT trap), 이후 통합 게이트(셸 7·py 10·selftest 10·dry-run 5) 전건 PASS 재실행.

# ⑫ 판정·평가 ("판단해봐 / 적합한가?")

**언제 쓰나**: 사용자가 가설·직관·대안을 던지며 판단을 구할 때. 산출물은 "근거 있는 판정" — 변경 실행이 아님(판정 후 지시 대기).

## 프롬프트

```
{{질문/가설}}을 판정하라.
1. 판정 기준을 먼저 명시 — 무엇이 "맞다/적합하다"의 기준인가 (품질? 비용? 실험 유효성? 안전?)
2. 증거 수집: 실측 > 코드 실물 > 문서 > 추론 순으로 — 이 세션/프로젝트의 실제 데이터를 우선
3. 사용자 직관의 맞는 부분과 조심할 부분을 분리해서 인정하라 — "절반 맞음"을 정직하게
   (전부 긍정도 전부 반박도 대체로 틀린 프레임이다)
4. 트레이드오프 표 (대안이 있으면)
5. 결론 + 적용 범위 한정: 이 판정이 유효한 조건과 뒤집힐 조건(어떤 데이터가 나오면 재고하나)
6. 판정만 요구받았으면 실행하지 마라 — 원복 비용을 만들지 않는다
```

**실증**: "8b든 70B든 상관없지 않나" → SVI 실측(0.75 vs 0.0)으로 판정 + "균일 모델이면 델타 보존" 반론 수용, "HTML 덱이 낫던데" → QA 결함 10건 대조로 채택+편집용 예외 한정, "50m 반경 넓지 않나" → 판단만 요구였는데 수정했다가 원복(실패 사례 — 6번 규칙의 기원).
